## 오늘 공부한 것
Next.js 라우팅을 파다 보니 라우트 종류가 너무 많아서 어지러웠다(`[]`, `[...]`, `()`, `@`, `(.)` ...). 한 번에 지도처럼 정리해두면 나중에 헷갈릴 때 이거 하나만 보면 될 것 같아서 정리.

## 내가 가졌던 질문
- 커스텀 앱(`_app`)이 뭐고 왜 쓰나
- `_document`은 `_app`이랑 뭐가 다르고 왜 필요한가
- 동적 경로 `[]` / `[...]` / `[[...]]` 차이
- 레이아웃을 어떻게 나누나, 중첩 레이아웃은 뭔가
- `()` 라우트 그룹 / `@` 병렬 라우트 / `(.)` 인터셉트 라우트 차이와 용도

## 큰 그림 먼저 — Pages Router vs App Router
많은 개념이 두 라우터로 갈린다. 지금 배우는 Pages Router는 예전 방식(여전히 지원), App Router가 요즘 기본.

| | Pages Router | App Router |
|---|---|---|
| 폴더 | `pages/` | `app/` |
| 공통 래퍼 | `_app.tsx` | `layout.js` |
| 페이지별/세그먼트별 레이아웃 | `getLayout` 패턴(수동) | 폴더별 `layout.js`(자동 중첩) |
| 라우트 그룹/병렬/인터셉트 | 없음 | 있음 (`()`,`@`,`(.)`) |

## 특수 파일: `_app` / `_document`
`_` 로 시작 = 주소가 아닌 특수 파일.

| | `_app.tsx` | `_document.tsx` |
|---|---|---|
| 역할 | 모든 페이지 **내용**을 감싸는 공통 래퍼 | 바깥 **HTML 뼈대**(`<html>/<body>/<head>`) |
| 둔다 | 전역 CSS, 공통 레이아웃, 전역 Provider | `<html lang>`, 폰트 링크, CSS-in-JS SSR 세팅 |
| 렌더 시점 | 페이지 이동마다 관여 | 서버에서 최초 1번만 |
| 인터랙션 | 넣을 수 있음 | 못 넣음(뼈대라서) |

- `_app`은 `<Component {...pageProps} />`로 지금 페이지를 그림. `Component`=현재 페이지, `pageProps`=그 데이터.
- `_document`은 대부분 필요 없음. `<html lang="ko">`나 CSS-in-JS SSR(첫 화면 스타일 깨짐=FOUC 방지) 같이 **HTML 뼈대 자체를 손볼 때만** 만든다.
- 참고: CSS-in-JS(styled-components 등)를 SSR로 쓸 때만 FOUC가 생김. 번쩍임을 없애는 건 `_document`의 SSR 세팅이고, CSS-in-JS는 오히려 원인 쪽. DB/데이터랑은 무관(스타일 문제).

## 동적 경로: `[]` / `[...]` / `[[...]]`
대괄호 = 그 자리가 유동적으로 바뀜. 고정 폴더(대괄호 없음)는 파라미터에 안 잡히고, 대괄호 세그먼트만 파라미터로 잡힌다.

| 표기 | 뜻 | 예 (`search` 폴더) | 파라미터 |
|---|---|---|---|
| `[id]` | 동적 **한 칸** | `/products/42` | `id: '42'` |
| `[...slug]` | catch-all, **여러 칸**(1칸 이상) | `/search/food/drinks` | `['food','drinks']` |
| `[[...slug]]` | optional catch-all, **0칸도**(베이스 포함) | `/search` 도 잡음 | `undefined` |

- `[id]` vs `[...]` 차이 = "바뀌는 칸이 하나냐 여러 개냐"(둘 다 동적).
- `[...]` vs `[[...]]` 차이 = "베이스 경로(`/search` 딱 하나)까지 잡느냐". 두 겹이 포함.
- 앞 폴더(`search`)는 고정이라 항상 파라미터에서 빠짐. catch-all은 그 뒤만 잡는다.
- **쓰는 곳**: 경로 깊이가 유동적이고 미리 폴더로 못 만들 때(문서 사이트, CMS 페이지, 깊은 카테고리). 깊이가 고정이면 그냥 `[id]`.

## 레이아웃
"공통이면 레이아웃." 여러 페이지가 같이 쓰는 **고정 틀**(헤더/푸터/사이드바). 그 안에 바뀌는 **콘텐츠**(`{children}`)가 들어간다. (레이아웃 ≠ 콘텐츠, 반대말.)

```jsx
function RootLayout({ children }) {
  return (<><Header />{children}<Footer /></>);  // children 자리에 페이지가 들어옴
}
```

- **범위로 나눈다**: 전체 공통 → 최상위 레이아웃 / 특정 섹션만 → 그 섹션 레이아웃.
- **중첩 레이아웃**: 레이아웃 안에 레이아웃. 헤더/푸터(전체) 안에 사이드바(대시보드만)를 겹침.
- **Pages Router**: 페이지별 레이아웃은 `getLayout` 패턴(각 페이지가 자기 레이아웃 선언 → `_app`이 불러 감쌈).
- **App Router**: 폴더별 `layout.js`가 폴더 깊이대로 자동 중첩.
- **HOC와의 관계**: 레이아웃=공통 UI 감싸 재사용, HOC=공통 로직 감싸 재사용. "감싸서 재사용"이라는 뿌리는 같음.
- 실무 감각: 처음부터 완벽히 나누려 말고, **반복이 실제로 보이면 그때 레이아웃/컴포넌트로 뽑는다.**

## 헷갈리는 3형제 — URL에 안 나타나는 폴더
`()`, `@`, `(.)` 셋 다 "폴더명이 URL에 안 들어감"이라 헷갈리는데, 하는 일은 완전히 다르다. (모두 App Router)

| | 표기 | 하는 일 | 한 화면에 |
|---|---|---|---|
| 라우트 그룹 | `(name)` | 라우트 **묶기/레이아웃 분리** | 페이지 **1개**(한 번에 하나) |
| 병렬 라우트 | `@name` | 여러 페이지 **동시 렌더**(슬롯) | **여러 개** 동시 |
| 인터셉트 | `(.)name` | **이동 방식**에 따라 다른 화면 | (모달 등) |

### 라우트 그룹 `()`
`(marketing)/about` → `/about` (그룹명은 URL에서 사라짐). 레이아웃은 그룹별로 나누되 URL은 깔끔하게. 여전히 한 번에 하나만 보임.

### 병렬 라우트 `@`
`@team`, `@analytics` 슬롯이 레이아웃에 props로 넘어와 **동시에** 렌더. 그냥 컴포넌트와 다른 점 = 각 슬롯이 **독립적인 로딩/에러(loading.js·error.js), 독립 라우팅, 조건부 교체**가 됨. 용례: 대시보드(영역별 독립 로딩), 역할별 화면, 모달.

### 인터셉트 라우트 `(.)`
같은 URL을 **어떻게 왔느냐**에 따라 다르게: 링크 클릭(soft) → 인터셉트 버전(모달) / 직접 접속·새로고침(hard) → 전체 페이지. 대표 용례 = **모달**(병렬 `@modal` + 인터셉트 `(.)` 결합). 일반 JS 모달과 달리 URL 있음 → 공유·뒤로가기·새로고침(전체페이지화)·SEO 다 됨.
- 점 개수 = 몇 단계 위를 가로채냐: `(.)` 같은 층 / `(..)` 한 층 위 / `(..)(..)` 두 층 / `(...)` 루트부터. (`../` 상대경로와 같은 개념)
- 함정: 레벨 셀 때 라우트 그룹 `()`·슬롯 `@`은 세그먼트로 안 침(URL 세그먼트 기준).

## 아직 헷갈리는 것 / 다음에 볼 것
- 인터셉트 라우트 실제 모달 구현 전체 코드 한 번 짜보기.
- App Router의 서버 컴포넌트 / 데이터 페칭(fetch, 캐시).
- `getLayout` 패턴에서 레이아웃 상태 유지가 정확히 어떻게 되는지.

## 한 줄 요약
Next.js 라우트는 "폴더/파일 규칙"으로 나뉜다 — `_app`(공통 래퍼)·`_document`(HTML 뼈대), `[]`계열(동적 경로: 한 칸/여러 칸/베이스 포함), 레이아웃(공통 틀·중첩), 그리고 URL에 안 나타나는 3형제 `()`(묶기)·`@`(동시 렌더)·`(.)`(이동 방식별 화면=모달). 종류는 많아도 "고정이냐 동적이냐 / 한 번에 하나냐 동시에 여러 개냐"로 가르면 정리된다.
