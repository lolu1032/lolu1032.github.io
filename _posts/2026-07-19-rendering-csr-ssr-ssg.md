## 오늘 공부한 것
CSR / SSR / SSG 세 가지 렌더링 방식의 흐름을 내 말로 설명해보고, 틀린 부분을 잡아가며 정리했다. 이어서 SSG가 배포되는 과정에서 나온 "CDN에 올린다"가 실제로 뭔지까지 봤다.

## 내가 가졌던 질문
- CSR/SSR/SSG 흐름을 내가 이해한 게 맞나?
- CSR은 처음에 HTML이 빈 화면인가? JS가 실행돼야 그려지나?
- SSR 파이프라인이 "브라우저 요청 → 서버가 DB에서 데이터 → 완성 HTML을 넘김"이 맞나?
- SSG는 배포된 정적 파일(html/css/js)을 바로 띄우는 형태인가?
- "CDN에 올린다"가 코드적으로 뭘 하는 건가?
- CDN을 불러오는 방법은?
- 티스토리·벨로그도 SSG인가?
- 고정이면 SSG, 실시간이면 SSR이라는데 — 그럼 SSG가 굳이 쓸모 있나?

## 정리 (Q&A 핵심)

### 세 방식을 가르는 기준: "HTML을 누가, 언제 만드느냐"

| | HTML 만드는 곳 | 만드는 시점 | 첫 화면 |
|---|---|---|---|
| CSR | 브라우저(JS) | 유저 접속 후 | 느림(빈 화면→렌더) |
| SSR | 서버 | 요청할 때마다 | 빠름 |
| SSG | 서버(빌드) | 미리(빌드 시) | 제일 빠름 |

### CSR (Client Side Rendering)
처음 받는 HTML은 **빈 껍데기**다.

```html
<body>
  <div id="root"></div>   <!-- 알맹이 없음 -->
  <script src="bundle.js"></script>
</body>
```

흐름:
1. 유저 요청 → 서버가 **빈 HTML + JS 번들** 보냄
2. 브라우저가 JS 실행 → **JS가 화면(DOM)을 그림** (화면이 채워지는 건 이 시점)
3. 데이터가 필요하면 → API 요청 → 응답 오면 그 부분을 다시 그림

즉 화면을 그리는 주체가 JS다. React가 대표적인 CSR. 단점은 JS를 받아 실행해야 화면이 보여서 **첫 화면이 느리고**, JS 실행 전엔 빈 div라 **검색엔진이 내용을 못 읽는(SEO)** 문제가 있다.

포인트: 화면을 **그리는 건 "JS 실행"**, 데이터를 **채우는 건 "API 응답"** — 둘은 별개 단계다.

### SSR (Server Side Rendering)
```
유저 → 브라우저 → 서버에 요청
         → 서버가 DB에서 데이터 받아옴
         → 서버가 데이터로 완성된 HTML을 만듦
         → 완성 HTML을 브라우저에 넘김 → 바로 보임
```

완성 HTML을 보낸 뒤 브라우저가 JS를 따로 받아 **하이드레이션**(이미 그려진 화면에 클릭·이벤트 기능을 입히는 마무리)을 한다. 안 하면 화면은 보이는데 버튼이 안 눌린다. SSR도 결국 JS를 받지만, **화면은 서버가 미리 그려서 보낸다**는 게 CSR과의 결정적 차이.

### SSG (Static Site Generation)
빌드 때 HTML을 미리 다 만들어 둔다.

```
1. 빌드: HTML을 전부 미리 생성 → 완성된 .html 파일들
2. 배포: 정적 파일(html/css/js)을 서버·CDN에 올림
3. 요청: 이미 만들어둔 HTML을 그냥 꺼내 줌
```

이 블로그(Jekyll + GitHub Pages)가 바로 SSG다. `git push` → GitHub Actions가 마크다운을 HTML로 빌드(~1분) → 정적 파일을 GitHub Pages(CDN)에 배포. 유저가 접속할 때 만드는 게 아니라 push 때 미리 만들어 둔다.

주의: 미리 만들어 둔 거라 **모든 방문자가 같은 HTML**을 본다. 블로그·문서처럼 고정 내용에 적합하고, 유저별로 다른 화면엔 부적합. 내용을 바꾸려면 **다시 빌드→재배포**.

SSR vs SSG 한 줄: 같은 완성 HTML이라도 **SSR은 요청마다 그 자리에서 만들고, SSG는 미리 만든 걸 꺼내 준다.**

### "CDN에 올린다"가 코드적으로 뭐냐
빌드하면 그냥 **파일 폴더**가 나온다 (`dist/`, Jekyll은 `_site/`).

```
dist/
├── index.html
├── posts/.../index.html
└── assets/style.css, main.js
```

"CDN에 올린다" = 이 폴더를 **원격 서버로 복사/업로드**하는 것. 서버 프로그램이 아니라 완성된 파일을 복사하는 거다.

```bash
aws s3 sync ./dist s3://my-blog-bucket   # S3 + CloudFront
netlify deploy --prod --dir=dist          # Netlify
vercel --prod                             # Vercel
```

이 블로그는 GitHub Actions가 `jekyll build`로 `_site/`를 만들고 그 폴더를 GitHub Pages(내부적으로 Fastly CDN)에 자동 업로드한다.

CDN = 전 세계 여러 지역에 흩어진 서버들. 파일을 올리면 각 지역 서버에 **복사본을 뿌려둔다(캐싱)**. 그래서 접속하면 **가장 가까운 서버**가 응답 → 빠르다. SSG가 빠른 이유가 여기서 완성된다(정적 파일이라 아무 서버에나 복사해둘 수 있음).

### CDN을 불러오는 방법 (두 가지 의미)

**의미 1) 남의 라이브러리를 CDN에서 불러오기** — 제일 흔한 의미
```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/bootstrap@5.3.0/dist/css/bootstrap.min.css">
<script src="https://cdn.jsdelivr.net/npm/axios@1.6.0/dist/axios.min.js"></script>
```
`src`/`href`에 CDN 주소를 박으면 브라우저가 그 URL에서 파일을 받아온다. 내가 보관 안 해도 되고 빠르지만, 그 CDN이 죽으면 못 불러오는 단점.

**의미 2) 내가 배포한 사이트가 CDN에서 불러와지는 것**
```
사용자가 도메인 입력 → DNS가 "이 도메인은 CDN 담당" 안내
   → 브라우저가 가장 가까운 CDN 서버로 요청 → 미리 저장된 HTML 응답
```
별도 코드 없이 **DNS가 알아서** 가까운 CDN 서버로 라우팅한다.

### 티스토리·벨로그는 SSG인가?
아니다. 이 둘은 **SSR/DB 기반**이다. 가르는 기준은 "내용을 어디에 저장하고, HTML을 언제 만드느냐".

| | 내용 저장 | HTML 만드는 시점 | 발행 방식 |
|---|---|---|---|
| 내 블로그 (SSG) | 파일(.md) | 빌드 때 미리 | push → 빌드 필요 |
| 티스토리/벨로그 (SSR) | 서버 DB | 요청 올 때마다 | 에디터에서 즉시 |

체감 증거: 티스토리는 글 쓰면 빌드 없이 **바로** 뜨고, 내 GitHub Pages 블로그는 push 후 ~1분 빌드를 기다려야 뜬다. (벨로그는 React/Next.js 서버 렌더링, 티스토리는 카카오 서버 렌더링 CMS.)

단, SEO는 **SSG·SSR 둘 다 좋다.** 봇이 완성 HTML을 받는다는 점은 같기 때문. 불리한 건 빈 HTML을 주는 **CSR**뿐.

### 고민: 그럼 SSG가 굳이 쓸모 있나?
"DB 들어가면 SSR"이라 생각했는데, 정리하니 SSG도 확실한 쓸모가 있었다.

**SSG가 SSR보다 나은 점**
1. 속도 — 미리 만든 파일이라 제일 빠름 (SSR은 요청마다 DB조회+렌더링 시간 붙음)
2. 서버 비용/부담 없음 — 파일만 서빙하면 되니 **서버가 아예 없어도 됨**. 내 블로그가 GitHub Pages에서 공짜인 이유.
3. 안정성 — 트래픽 몰려도 정적 파일이라 잘 안 죽음(CDN이 감당)
4. 보안 — DB·서버 로직이 없어 SQL 인젝션 같은 공격이 애초에 성립 안 됨

**"DB=SSR"은 정확히는 아니다** — SSG도 데이터를 쓸 수 있다. 차이는 *언제* 가져오냐.
```
SSR: 요청마다 DB에서 가져와 HTML 생성   (실시간)
SSG: 빌드 때 한 번 가져와 HTML 미리 생성 (고정)
```
그래서 진짜 기준은 "DB를 쓰냐"가 아니라 **"내용이 요청마다/유저마다 바뀌냐"**.
- 모두 같고 자주 안 바뀜 → SSG (블로그·문서·랜딩·소개)
- 유저별로 다르거나 실시간(마이페이지, 댓글, 좋아요, 재고, 시세) → SSR/CSR

**현실은 섞어 쓴다** — Next.js 등은 페이지별로 SSG/SSR/CSR을 골라 씀. 안 바뀌는 페이지는 SSG, 바뀌는 페이지는 SSR. (SSG인데 주기적으로 다시 빌드해 어느 정도 최신화하는 **ISR**도 있음 — 이름만 메모.)

결론 감각: **고정적이면 SSG, 실시간/유저별이면 SSR.** 둘은 경쟁이 아니라 역할 분담.

## 아직 헷갈리는 것 / 다음에 볼 것
- 하이드레이션 과정을 더 구체적으로(왜 서버 HTML과 클라 JS가 안 맞으면 에러 나는지 등).
- Next.js 같은 프레임워크에서 CSR/SSR/SSG/ISR을 페이지별로 섞어 쓰는 방식.
- CDN 캐시 무효화(내용 바꿨을 때 이전 캐시가 남는 문제) 처리법.

## 한 줄 요약
CSR/SSR/SSG는 "HTML을 누가·언제 만드느냐"의 차이(브라우저가 실행 후 / 서버가 요청마다 / 서버가 빌드 때)이고, SSG의 배포는 빌드로 나온 정적 파일을 CDN(전 세계 서버망)에 복사해 가까운 데서 서빙하는 것.
